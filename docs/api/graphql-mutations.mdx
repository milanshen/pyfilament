Pyfilament exposes two GraphQL mutations: `cancelTaskRun` to stop a running or pending task, and `runTask` to trigger a task type from the API without calling Python directly.

---

## cancelTaskRun

Cancel a running or pending task run. Pyfilament updates the run's state to `cancelled` and sends a cancellation signal to the executing worker.

**Arguments:**

<ParamField body="id" type="ID">
  The database ID of the task run to cancel.
</ParamField>

<ParamField body="taskUuid" type="String">
  The UUID of the task run to cancel.
</ParamField>

Provide either `id` or `taskUuid` — at least one is required.

**Returns:** `TaskRun` — the task run after cancellation has been initiated. The `state` field reflects the updated state.

```graphql
mutation {
  cancelTaskRun(taskUuid: "abc123") {
    taskUuid
    state
    stateSince
  }
}
```

<Note>
  Tasks that have already reached a [terminal state](/api/task-states) will not be affected.
</Note>

---

## runTask

Trigger a task type to execute with a given set of parameters. Pyfilament creates a new task run and starts execution immediately if a worker is available, then returns the run.

**Arguments:**

<ParamField body="taskTypeId" type="ID" required>
  The database ID of the task type to run.
</ParamField>

<ParamField body="parametersJson" type="String" required>
  A JSON-encoded string of keyword arguments to pass to the task function. Keys must match the function's parameter names.
</ParamField>

**Returns:** `TaskRun` — the newly created task run.

```graphql
mutation {
  runTask(
    taskTypeId: "1",
    parametersJson: "{\"order_id\": \"ord-999\"}"
  ) {
    taskUuid
    state
    createdAt
  }
}
```

`parametersJson` is a JSON string of the task function's keyword arguments. For example, if your task is defined as:

```python
@task
async def process_order(order_id: str) -> dict:
    ...
```

then `parametersJson` should be `"{\"order_id\": \"ord-999\"}"`.

The `runTask` mutation sets `start_immediately: true` automatically, so execution begins as soon as a worker picks it up.

<Note>
  If no worker is running for the target task type, `runTask` enqueues the run but it will not execute until a worker starts. Poll `getTaskRun` to check when the run transitions out of `created` state.
</Note>
