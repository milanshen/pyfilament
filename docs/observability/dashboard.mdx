Pyfilament includes a built-in web dashboard for monitoring your tasks. Browse all task types, inspect individual runs, view execution timelines, read structured logs, and cancel running tasks — all from the browser. The dashboard has two parts: a **FastAPI** backend that serves the GraphQL and REST APIs, and a **React** frontend that talks to it.

## Start the dashboard

The dashboard reads task data from the same database your tasks write to, so point both processes at that database and at the Redis instance your tasks use. See [Installation](/installation) for the `FILAMENT_DB_URI` and `FILAMENT_REDIS_*` variables (the defaults — local SQLite and `localhost:6379` — need no configuration) and for applying the [database migrations](/installation#create-the-schema) that create the schema.

### Run the backend

Start the FastAPI app with Uvicorn from the repository root. It serves the GraphQL and REST APIs on port `5006`:

```bash
uv run uvicorn filament.api.main:app --host 0.0.0.0 --port 5006 --reload --reload-dir src/
```

`--reload` restarts the server when you edit files under `src/`; drop it for a non-development run.

### Run the frontend

The frontend is a Vite app in the `ui/` directory. Install its dependencies once, then start the dev server:

```bash
cd ui
npm install
npm run dev
```

It serves the dashboard on [http://localhost:3001](http://localhost:3001) and proxies `/graphql` and `/api` requests to the backend on port `5006`. Open that URL in your browser to use the dashboard.

<Tip>
  For a production-style build, run `npm run build` then `npm run preview` instead of
  `npm run dev`. Both serve on port `3001`.
</Tip>

## Dashboard pages

The dashboard is a React single-page application with five main pages:

1. **Task Types** (`/`) — lists all registered task types sorted by most-recent activity. Each row shows the task name, its latest run's state badge, and a human-readable timestamp.
2. **Task Type** (`/task-type/:taskTypeId`) — shows the most recent runs for a single task type (up to 99 runs), sorted by most-recently-updated first. Filter runs by state to find failures or long-running executions quickly.
3. **Task Run** (`/task-run/:taskRunId`) — the primary debugging view. Shows the full execution timeline, subtask tree, parameter and result JSON, and structured logs for a single run. Accepts either a numeric database ID or a task UUID.
4. **Task Type Stack** (`/task-type-stack/:taskTypeIds`) — explore runs across a call stack of task types. Pass a comma-separated list of task type IDs to visualise how runs flow through a chain of tasks.
5. **Compare Runs** (`/task-runs-compare/:taskRunIds`) — diff the parameters and results between two runs side by side. Pass two comma-separated run IDs to spot regressions or unexpected input changes.

## Task run timeline

The timeline view on the Task Run page renders all subtasks as nested horizontal bars on a Gantt-style chart, proportional to wall-clock duration. The root task occupies the top row; its children are indented below it, and their children below them. Click any bar to select that task run and open its detail panel on the right.

## Inspecting a run

The detail panel shows everything recorded for a selected task run:

- **State badge** — current state (`created`, `running`, `success`, `failure`, `cancelled`).
- **State since** — when the task entered its current state.
- **Heartbeat** — timestamp of the last heartbeat from the executing worker, useful for detecting stuck tasks.
- **Run count** — how many times this task has been attempted.
- **Parameters** — the input arguments serialised as JSON.
- **Result** — the return value serialised as JSON, populated once the task reaches a terminal state.
- **Logs** — the structured log entries emitted by `get_logger()` during execution (see the [Logging](/observability/logging) guide).

The logs panel beneath the detail panel shows all log entries for the selected run and its subtasks, sorted chronologically so you see a unified event stream across the entire execution tree.

<Tip>
  You can navigate to a task run directly by UUID: `/task-run/{uuid}`. The dashboard resolves the UUID to its numeric ID automatically.
</Tip>
